### 1.7 Food Preservation Batch Tracker

### Strategic Insight

NOMAD's feature breadth is genuinely unique — no single competitor covers even half of its modules. The closest dedicated prepper tools (Ultimate Prep System, PrepStack, Preppr) are cloud-dependent mobile apps focused almost exclusively on inventory. The closest self-hosted platforms (Grocy, Home Assistant, farmOS) each cover one or two of NOMAD's domains deeply but don't attempt the others. This is NOMAD's moat.

## Design Principles for New Features

1. **Offline-first** — Every feature must work without internet. Network features are enhancements, not requirements.
2. **Data sovereignty** — All data stays on the user's machine. No telemetry, no cloud dependency, no accounts.
3. **Print-friendly** — Every data view should have a print-optimized version. Binders are standard in the community.
4. **Modular** — Features should be independently usable. Don't force dependencies between modules.
5. **Low-resource compatible** — Must run on Raspberry Pi, old laptops, portable hardware. No Electron bloat.
6. **Secure by default** — Sensitive data (locations, medical, financial) encrypted. API endpoints authenticated.
7. **Import/export everything** — CSV, JSON, GPX, PDF, KML, CHIRP, vCard, ICS. No data lock-in ever.
8. **AI-optional** — AI features enhance but never gate core functionality. Everything works without Ollama.
9. **Federation-aware** — New modules should consider what data could be shared between trusted nodes.
10. **Accessible** — WCAG 2.1 AA compliance. Screen reader support. Keyboard navigable. High contrast mode.
11. **Composable** — Modules should feed data to each other. Inventory → Resource Planner → Shopping List → Readiness Score.
12. **Field-tested** — Features should work under stress: low light, gloved hands, time pressure, poor connectivity.
13. **Printable last resort** — If the device dies, printed reports should contain enough info to operate. Paper is the ultimate backup.
14. **Seasonal awareness** — Features should adapt to the time of year. Surface relevant checklists, adjust calculations, remind about seasonal tasks.
15. **Recovery-oriented** — Don't just plan for the event. Plan for after. Every module should support the return to normalcy.
16. **Community-multiplied** — One person's work should benefit the network. Content packs, shared templates, and federation amplify individual effort.
17. **Tactically informed** — Draw from ICS/NIMS, military planning, and SAR methodologies. These are battle-tested frameworks — adopt them, don't reinvent them.
18. **Scenario-connected** — Every feature should be able to answer "in scenario X, what does this data mean?" Connect inventory to scenarios, connect skills to roles, connect locations to plans.

## Design Principles (Updated)

1. **Offline-first** — Every feature must work without internet. Network features are enhancements, not requirements.
2. **Data sovereignty** — All data stays on the user's machine. No telemetry, no cloud dependency, no accounts.
3. **Print-friendly** — Every data view should have a print-optimized version. Binders are standard in the community.
4. **Modular** — Features should be independently usable. Don't force dependencies between modules.
5. **Low-resource compatible** — Must run on Raspberry Pi, old laptops, portable hardware. No Electron bloat.
6. **Secure by default** — Sensitive data (locations, medical, financial) encrypted. API endpoints authenticated.
7. **Import/export everything** — CSV, JSON, GPX, PDF, KML, CHIRP, vCard, ICS. No data lock-in ever.
8. **AI-optional** — AI features enhance but never gate core functionality. Everything works without Ollama.
9. **Federation-aware** — New modules should consider what data could be shared between trusted nodes.
10. **Accessible** — WCAG 2.1 AA compliance. Screen reader support. Keyboard navigable. High contrast mode.
11. **Composable** — Modules should feed data to each other. Inventory → Resource Planner → Shopping List → Readiness Score.
12. **Field-tested** — Features should work under stress: low light, gloved hands, time pressure, poor connectivity.
13. **Printable last resort** — If the device dies, printed reports should contain enough info to operate. Paper is the ultimate backup.
14. **Fail graceful** — No feature should crash or block others. Missing services degrade gracefully. Corrupt data recovers.
15. **Seasonal awareness** — Features should adapt to the time of year. Surface relevant checklists, adjust calculations, remind about seasonal tasks.
16. **Recovery-oriented** — Don't just plan for the event. Plan for after. Every module should support the return to normalcy.
17. **Community-multiplied** — One person's work should benefit the network. Content packs, shared templates, and federation amplify individual effort.
18. **Tactically informed** — Draw from ICS/NIMS, military planning, and SAR methodologies. These are battle-tested frameworks — adopt them, don't reinvent them.
19. **Scenario-connected** — Every feature should be able to answer "in scenario X, what does this data mean?" Connect inventory to scenarios, connect skills to roles, connect locations to plans.
20. **Regionally adapted** — No feature should assume a single climate, terrain, legal framework, or threat profile. Adapt to where the user actually lives.
21. **Multi-scale** — Features should work for a solo individual, a family, a pod of households, or a full federation network. Don't design only for one group size.
22. **Long-horizon** — Plan for 72 hours, 30 days, 1 year, and 10+ years. Short-term survival is only the beginning — sustainability requires a longer view.
